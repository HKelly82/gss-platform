# GSS Platform — setup and run guide

This repo builds the e-learning platform front-end: a Next.js app that renders the curriculum
content and deploys to Vercel. It mirrors the curriculum build's discipline — two gates (review,
then build), QA subagents, and a hard deploy gate.

## What's in here

```
gss-platform/
  CLAUDE.md                       the build contract (loaded every session)
  SETUP.md                        this file
  vercel.json                     framework + basic security headers
  .claude/
    settings.json                 ntfy Stop hook + read-only guard on content/ and design/
    commands/                     review-build, scaffold, build-component, build-route, qa, deploy
    agents/                       prototype-critic, content-contract, design-fidelity,
                                  a11y-auditor, content-integrity, build-health
  content/                        READ-ONLY — the curriculum content (you wire this; see step 2)
  design/                         READ-ONLY — the Claude Design language (you add this; see step 2)
  working/
    decisions/                    the build plan (generated)
    qa-reports/                   QA output (generated)
```

The Next.js app itself (`app/`, `components/`, `lib/`, `package.json`, etc.) is generated by
`/scaffold` — this scaffold is just the control layer.

## Step 1 — place the scaffold

Put this `gss-platform/` folder wherever you keep the project and open Claude Code in it (the
folder with `CLAUDE.md` is the root).

## Step 2 — wire the read-only sources

- **`content/`** — the curriculum content. Add the curriculum repo's `output/` as a git submodule
  so it stays read-only and versioned, e.g. `git submodule add <curriculum-repo-url> content-src`,
  then point `content/` at `content-src/output` (or sync `output/` into `content/`). The app reads
  the 218 markdown files from here.
- **`design/`** — the design language from Claude Design. Export its tokens and component inventory
  here, plus a `DESIGN-SPEC.md` describing the archetypes and their states. If you keep Figma as the
  source of truth, connect the Figma MCP and drop the exported tokens here as the mirror;
  `design-fidelity` reads the live file via MCP.

Delete the `.PUT-FILES-HERE` placeholders once each is filled. If either source is missing what a
task needs, the review gate reports it as a blocker rather than guessing.

## Step 3 — confirm the guard and hook

`.claude/settings.json` denies writes to `content/` and `design/`, and pings
`ntfy.sh/hmkel-claude-notifications` on Stop. Confirm the deny globs fire in your Claude Code version
(the syntax has drifted between releases) by asking Claude to edit a file under `content/` and
checking it is refused; the `CLAUDE.md` rule is the primary guard and the permission is backstop.

## Step 4 — connect Vercel

Connect the repo to Vercel (Git integration) so every push gets an automatic preview deploy — those
`vercel.app` URLs are shareable for Herd or DHSC review. Keep `/deploy` for on-demand preview or
production via the CLI. `vercel.json` sets the framework and basic security headers; content is
static (SSG), so no database or environment variables are needed for the platform.

## Step 5 — the run sequence

The loop is review -> approve -> build -> QA self-heals -> you clear flags, the same as the
curriculum build. Approve a plan by setting its first line to `status: approved`.

```
1. /review-build              -> working/decisions/build-plan.md, then approve it
2. /scaffold                   one-time: app, content loader, progress module, tokens, base layout
3. /build-component <type>     each renderer: multiple-choice, drag-drop, short-answer-reveal,
                               diagnostic-router, scenario-narrative, guided-content,
                               understanding-check, takeaway, reference-card
4. /build-route tier           the proof of concept: one full tier flow end to end -> STOP, review
5. /build-route <route>        the rest: module-overview, pathway-BA/DM/PM/SME, sme-exercise
6. /qa                         full chain across the app
7. /deploy preview             validate the shared URL
8. /deploy prod                when you're happy
```

The proof-of-concept gate at step 4 is deliberate: wire one complete tier — diagnostic with both
the skip and proceed paths, scenario, guided content, understanding check with feedback, takeaway —
before building the rest, so you catch a structural problem once rather than across every module.

## Step 6 — QA flags

Every build runs its QA agents and self-heals up to three passes. Anything unresolved is listed in
the run summary and under `working/qa-reports/`. Fix it, then re-run `/qa` or the specific build.
`build-health` and `a11y-auditor` are hard gates on `/deploy` — the app will not ship while either
fails.

## The two failure modes this exists to catch

1. **The build quietly breaking the learning design** — re-ordering components, mis-wiring the
   diagnostic routing, or revealing a model answer early. `content-contract` guards this against the
   curriculum's `DESIGNER-HANDOVER.md`.
2. **Shipping an inaccessible platform** for a product whose subject is accessibility.
   `a11y-auditor` guards this to WCAG 2.2 AA as a hard deploy gate.

Each QA agent runs as a separate subagent with fresh context, on purpose: the context that wrote the
code is the wrong one to audit it.

## Customising

- **Model routing** is set per agent in each agent file. Adjust to taste and budget.
- **The must-not-change list** lives in both `CLAUDE.md` and `content-contract.md` — keep them in
  sync if it changes.
- **Add a check** by dropping a new agent in `.claude/agents/` and wiring it into the relevant
  command's QA chain.
